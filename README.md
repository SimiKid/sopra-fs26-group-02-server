# SoPra FS26 — Wizard Battle Server

Backend for a two-player wizard battle game. Built for the UZH SoPra FS26 course.

## Overview

Players register, create or join a game session by code, pick a wizard class and three attacks, and then take turns dealing damage until one wizard's HP reaches zero. Real-time updates are pushed to the frontend over WebSocket (STOMP). Live weather at the arena location (fetched from OpenWeather) modifies damage.

**Game flow:** register → login → create/join session (6-char code) → pick wizard + attacks → battle (turn-based, 30s auto-attack timer) → view history / offer rematch.

## Tech stack

- Java 17, Spring Boot 4
- Spring Web MVC + WebSocket (STOMP)
- Spring Data JPA + Hibernate, H2 (local) / PostgreSQL 18 on Cloud SQL (production)
- SpringDoc OpenAPI
- JUnit 5, Mockito (unit + `@SpringBootTest` integration tests)
- Gradle, Docker, GitHub Actions → Docker Hub

## Prerequisites

- JDK 17
- An [OpenWeather](https://openweathermap.org/api) API key (the game falls back to a random-weather arena if the call fails, so tests/local runs work without one)

## Configuration

Secrets live in `local.properties` (gitignored, auto-loaded by `build.gradle`):

```properties
openweather.api.key=<your-key>
```

## Running

```bash
./gradlew bootRun           # start server on http://localhost:8080
./gradlew build --continuous -xtest   # hot rebuild on file change (pair with bootRun)
./gradlew test              # run all tests
./gradlew test --tests '*IntegrationTest'   # integration tests only
```

## Useful local URLs

- Swagger UI: `http://localhost:8080/swagger-ui.html`
- H2 console: `http://localhost:8080/h2-console` (JDBC URL `jdbc:h2:mem:testdb`, user `sa`, no password)

## Project layout

```
src/main/java/ch/uzh/ifi/hase/soprafs26/
  controller/    REST + WebSocket endpoints
  service/       Business logic (UserService, GameSessionService, BattleService, ...)
  entity/        JPA entities (User, GameSession, Player, Battle)
  repository/    Spring Data repositories
  rest/dto/      Request/response DTOs + MapStruct mappers
  interceptor/   AuthInterceptor (token-based)
  config/        Spring MVC + WebSocket configuration

```

## Tests

- **Unit tests** — `*Test.java`: services mocked with Mockito, controllers with `@WebMvcTest` + MockMvc.
- **Integration tests** — `*IntegrationTest.java`: `@SpringBootTest` + `@Transactional`, real H2 and real Spring wiring. Cover the user lifecycle, game-session state machine, and login/token/logout round-trip.

## Deployment

Pushes to `main` trigger a GitHub Actions workflow that builds and publishes a Docker image to Docker Hub. Run locally with:

```bash
docker pull <dockerhub_username>/<dockerhub_repo_name>
docker run -p 8080:8080 <dockerhub_username>/<dockerhub_repo_name>
```

## Database Setup

### Local Development

The app uses an **H2 in-memory database** by default. No setup required — just run the app locally and it works automatically.

### Production (Google Cloud SQL)

The deployed app connects to a **PostgreSQL 18** instance on Google Cloud SQL (europe-west6).

The following environment variables must be set (in `app.yaml`, never committed to git):

| Variable | Description |
|---|---|
| `DB_URL` | JDBC connection string for Cloud SQL |
| `DB_USER` | Database username (default: `postgres`) |
| `DB_PASSWORD` | Database password |
| `SPRING_PROFILES_ACTIVE` | Must be set to `prod` |

The schema is auto-generated by Hibernate on first startup (`ddl-auto=update`).

### Resetting the Database

To wipe all data and start fresh, connect to the Cloud SQL instance via Cloud SQL Studio in the Google Cloud Console and run:

```sql
DROP SCHEMA public CASCADE;
CREATE SCHEMA public;
```

Then redeploy — Hibernate will recreate all tables automatically.
